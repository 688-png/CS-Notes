## 1. The role of build tools

Building a project usually includes dependency management, testing, compilation, packaging, publishing and other processes. Build tools can automate these operations, thereby reducing these tedious tasks for us.

The dependency management provided by the build tool can automatically handle dependencies. For example, if a project requires dependency A, and A depends on B, then the build tool can help us import B, without the need for us to manually find and import it.

In Java projects, the packaging process usually involves packaging the project into a Jar package. In the absence of a build tool, we need to use command line tools or IDE to package manually. The publishing process usually involves uploading the Jar package to the server.

## 2. Java mainstream build tools

Ant has compilation, testing and packaging functions. Maven, which appeared later, added dependency management functions based on Ant's functions, and the latest Gradle added support for the Groovy language based on Maven's functions.

The difference between Gradle and Maven is that it uses Groovy, a domain-specific language (DSL), to manage build scripts, instead of XML, a markup language. Because if the project is large, XML can easily become bloated.

For example, to introduce Junit into the project, the Maven code is as follows:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
  xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 http://maven.apache.org/xsd/maven-4.0.0.xsd">
   <modelVersion>4.0.0</modelVersion>
 
   <groupId>jizg.study.maven.hello</groupId>
   <artifactId>hello-first</artifactId>
   <version>0.0.1-SNAPSHOT</version>

   <dependencies>
          <dependency>
               <groupId>junit</groupId>
               <artifactId>junit</artifactId>
               <version>4.10</version>
               <scope>test</scope>
          </dependency>
   </dependencies>
</project>
```

Gradle only requires a few lines of code:

```java
dependencies {
    testCompile "junit:junit:4.10"
}
```

## 3. Maven

### Overview

A Project Object Model (POM) file is provided to manage the construction of the project.

### Warehouse

The search order of warehouses is: local warehouse, central warehouse, remote warehouse.

- The local warehouse is used to store the project’s dependent libraries;
- The central repository is the default location for downloading dependent libraries;
- Remote warehouse, because not all dependent libraries are in the central
warehouse, or the access speed of the central warehouse is very slow, the remote warehouse is a supplement to the central warehouse.

### POM

POM stands for Project Object Model, which is an XML file saved in the pom.xml file in the root directory of the project.

```xml
<dependency>
    <groupId>junit</groupId>
    <artifactId>junit</artifactId>
    <version>4.12</version>
    <scope>test</scope>
</dependency>
```

[groupId, artifactId, version, packaging, classifier] is called the coordinates of a project, where groupId, artifactId, and version must be defined, packaging is optional (default is Jar), and classifier cannot be defined directly and needs to be used in conjunction with a plug-in.

- groupId: project group ID, must be globally unique;
- artifactId: project ID, which is the project name;
- version: project version;
- packaging: project packaging method.

### Dependency principle

#### 1. Dependency path shortest first principle

```html
A -> B -> C -> X(1.0)
A -> D -> X(2.0)
```
Since X(2.0) has the shortest path, use X(2.0).

#### 2. Declaration order precedence principle

```html
A -> B -> X(1.0)
A -> C -> X(2.0)
```

The first one declared in the POM takes precedence. If B is declared first for the above two dependencies, X(1.0) will be used last.

#### 3. Overwrite priority principle

Dependencies declared in the child POM take precedence over dependencies declared in the parent POM.

### Resolve dependency conflicts

Find the Jar package version loaded by Maven, use `mvn dependency:tree` to view the dependency tree, and adjust the declaration order of dependencies in the POM file according to the dependency principle.
