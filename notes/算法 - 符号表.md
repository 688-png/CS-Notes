## Preface

Symbol Table is a data structure that stores key-value pairs and can support fast search operations.

The symbol table is divided into two types: ordered and unordered. The ordered symbol table mainly refers to supporting min(), max() and other operations based on the size relationship of keys.

The keys of the ordered symbol table need to implement the Comparable interface.

```java
public interface UnorderedST<Key, Value> {

    int size();

    Value get(Key key);

    void put(Key key, Value value);

    void delete(Key key);
}
```

```java
public interface OrderedST<Key extends Comparable<Key>, Value> {

    int size();

    void put(Key key, Value value);

    Value get(Key key);

    Key min();

    Key max();

    int rank(Key key);

    List<Key> keys(Key l, Key h);
}
```

### 2. Binary search to implement ordered symbol table

Use a pair of parallel arrays, one to store keys and one to store values.

The rank() method of binary search is crucial because it knows where the key is when it is in the table and where to insert the new key when it is not.

Binary search requires at most logN+1 comparisons, and the search operation of the symbol table implemented using binary search requires at most a logarithmic level. But the insertion operation requires moving array elements, which is linear level.

```java
public class BinarySearchOrderedST<Key extends Comparable<Key>, Value> implements OrderedST<Key, Value> {

    private Key[] keys;
    private Value[] values;
    private int N = 0;

    public BinarySearchOrderedST(int capacity) {
        keys = (Key[]) new Comparable[capacity];
        values = (Value[]) new Object[capacity];
    }

    @Override
    public int size() {
        return N;
    }

    @Override
    public int rank(Key key) {
        int l = 0, h = N - 1;
        while (l <= h) {
            int m = l + (h - l) / 2;
            int cmp = key.compareTo(keys[m]);
            if (cmp == 0)
                return m;
            else if (cmp < 0)
                h = m - 1;
            else
                l = m + 1;
        }
        return l;
    }

    @Override
    public List<Key> keys(Key l, Key h) {
        int index = rank(l);
        List<Key> list = new ArrayList<>();
        while (keys[index].compareTo(h) <= 0) {
            list.add(keys[index]);
            index++;
        }
        return list;
    }

    @Override
    public void put(Key key, Value value) {
        int index = rank(key);
        // If an existing node key is found, update the value of this node to value.
        if (index < N && keys[index].compareTo(key) == 0) {
            values[index] = value;
            return;
        }
        // Otherwise, to insert a new node into the array, you need to first move all the elements after the insertion position backward by one position.
        for (int j = N; j > index; j--) {
            keys[j] = keys[j - 1];
            values[j] = values[j - 1];
        }
        keys[index] = key;
        values[index] = value;
        N++;
    }

    @Override
    public Value get(Key key) {
        int index = rank(key);
        if (index < N && keys[index].compareTo(key) == 0)
            return values[index];
        return null;
    }

    @Override
    public Key min() {
        return keys[0];
    }

    @Override
    public Key max() {
        return keys[N - 1];
    }
}
```

## Binary search tree

**Binary tree** is an empty link, or a node with two left and right links, each link points to a sub-binary tree.

**Binary Search Tree** (BST) is a binary tree, and the value of each node is greater than or equal to the value of all nodes in its left subtree and less than or equal to the value of all nodes in its right subtree.

An important property of BST is that its in-order traversal results are sorted in ascending order.

### 1.get()

- If the tree is empty, the search misses;
- If the key being searched is equal to the key of the root node, the search hits;
- Otherwise, search recursively in the subtree: if the key being searched is smaller, it is searched in the left subtree, if it is larger, it is searched in the right subtree.

```java
@Override
public Value get(Key key) {
    return get(root, key);
}

private Value get(Node x, Key key) {
    if(x==null)
        return null;
    int cmp = key.compareTo(x.key);
    if (cmp == 0)
        return x.val;
    else if (cmp < 0)
        return get(x.left, key);
    else
        return get(x.right, key);
}
```

### 2. put()

When the inserted key does not exist in the tree, a new node needs to be created, and the link of the upper node is updated to point to the node so that the node is correctly linked to the tree.

```java
        return new Node(key, value, 1);
    int cmp = key.compareTo(x.key);

        x.val = value;
    else if (cmp < 0)
```

### 4. floor()

floor(key): the largest key less than or equal to the key

- If the key is smaller than the key of the root node, then floor(key) must be in the left subtree;
- If the key is greater than the key of the root node, you need to first determine whether floor(key) exists in the right subtree, and return if it exists, otherwise the root node is floor(key).

```java
public Key floor(Key key) {
    Node x = floor(root, key);
    if(x==null)
        return null;
    return x.key;
}

private Node floor(Node x, Key key) {
    if(x==null)
        return null;
    int cmp = key.compareTo(x.key);
    if (cmp == 0)
        return x;
    if (cmp < 0)
        return floor(x.left, key);
    Node t = floor(x.right, key);
    return t != null ? t : x;
}
```

### 5. rank()

rank(key) returns the ranking of key.

- If the key is equal to the key of the root node, return the number of nodes in the left subtree;
- If less than, recursively calculate the ranking in the left subtree;
- If greater, recursively calculate the rank in the right subtree, plus the number of nodes in the left subtree, plus 1 (the root node).

```java
@Override
public int rank(Key key) {
    return rank(key, root);

        return 0;

    if (cmp == 0)
        return size(x.left);
    else if (cmp < 0)
        return rank(key, x.left);
    else
```

## 2-3 Search tree

2-3 search trees introduce 2-nodes and 3-nodes in order to balance the tree. A perfectly balanced 2-3 search tree should have all empty links at the same distance from the root node.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/1097658b-c0e6-4821-be9b-25304726a11c.jpg" width="160px"/> </div><br>

### 1. Insert operation

The insertion operation is very different from that of BST. The insertion operation of BST first performs a miss search and then inserts the node into the corresponding empty link. But if the 2-3 search tree does this, it will destroy the balance. It inserts new nodes into leaf nodes.

Depending on the type of leaf node, there are different processing methods:

- If inserted into a 2-node, then directly combine the new node and the original node to form a 3-node.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/0c6f9930-8704-4a54-af23-19f9ca3e48b0.jpg" width="350"/> </div><br>

- If inserted into a 3-node, a temporary 4-node will be generated. It is necessary to split the 4-node into three 2-nodes and move the middle 2-node to the upper node. If the move-up operation continues to generate temporary 4-nodes, splitting and move-up will continue until there are no temporary 4-nodes.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/7002c01b-1ed5-475a-9e5f-5fc8a4cdbcc0.jpg" width="460"/> </div><br>

### 2. Properties

2-3 The transformations of the search tree insertion operation are all local. There is no need to modify or check other parts of the tree except related nodes and links, and these local transformations will not affect the global ordering and peace of the tree.
balance.

The complexity of the search and insertion operations of the 2-3 search tree has nothing to do with the insertion order. In the worst case, the number of nodes visited by the search and insertion operations must not exceed logN. A 2-3 search tree containing 1 billion nodes only needs to visit a maximum of 30 nodes to perform any search and insertion operations.

## Red-black tree

The red-black tree is a 2-3 search tree, but it does not need to define 2-nodes and 3-nodes separately. Instead, it adds colors to the nodes on top of the ordinary binary search tree. If the link color pointing to a node is red, then this node and the upper node represent a 3-node, while black is a normal link.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/f1912ba6-6402-4321-9aa8-13d32fd121d1.jpg" width="240"/> </div><br>

Red-black trees have the following properties:

- The red links are all left links;
- Perfect black balance, that is, there are the same number of black links on the path from any empty link to the root node.

When drawing a red-black tree, you can draw the red links flat.

### 1. Rotate left

Because legal red links are all left links, if a right link appears as a red link, then a left rotation operation is required.

From the perspective of the 2-3 tree, it means moving the middle node to the upper node.

First insert a node into the correct position according to the binary search tree method, and then perform the following color operations:

- If the right child node is red and the left child node is black, perform a left rotation;
- If the left child node is red and the left child node of the left child node is also red, perform a right rotation;
- If the left and right child nodes are both red, perform color conversion.

### 5. Analysis

The height of a red-black tree of size N will not exceed 2logN. The worst case scenario is that of its corresponding 2-3 tree, the leftmost path nodes are all 3-nodes and the rest are 2-nodes.

Most operations on red-black trees take logarithmic time.

## Hash table

A hash table is similar to an array, and the hash value of the hash table can be regarded as the index value of the array. Accessing a hash table is as fast as accessing array elements, and it can perform lookup and insertion operations in constant time.

Since the size relationship of the keys cannot be known through the hash value, the hash table cannot implement ordered operations.

### 1. Hash function

For a hash table of size M, the hash function can convert any key into a positive integer within [0, M-1], and the positive integer is the hash value.

There are collisions in hash tables, that is, two different keys may have the same hash value.

The hash function should satisfy the following three conditions:

- Consistency: Equal keys should have equal hash values. If two keys are equal, it means that the values returned by calling equals() are equal.
- Efficiency: The calculation should be simple. If necessary, the hash value can be cached and returned directly when the hash function is called.
- Uniformity: The hash values ​​of all keys should be evenly distributed between [0, M-1]. If this condition is not met, many conflicts may occur, resulting in a decrease in the performance of the hash table.

The division-remainder method can hash an integer between [0, M-1]. For example, for a positive integer k, calculating k%M can get a hash value between [0, M-1]. Note that M is preferably a prime number, otherwise all the information contained in the key cannot be exploited. For example, if M is 10<sup>k</sup>, then only the last k bits of the key can be used.

For other numbers, you can convert them into integer form and then use the division with remainder method. For example, for floating point numbers, you can convert their binary form to an integer.

For multi-part types, each part needs to have a hash value calculated, and these hash values ​​are equally important. To achieve this, think of the type as an R-based integer, with each part having a different weight.

For example, the hash function for strings is implemented as follows:

```java
int hash = 0;
for (int i = 0; i < s.length(); i++)
    hash = (R * hash + s.charAt(i)) % M;
```

For another example, the hash function of a custom class with multiple members is as follows:

```java
int hash = (((day * R + month) % M) * R + year) % M;
```

R is usually taken as 31.

hashCode() in Java implements the hash function, but uses the memory address value of the object by default. When using hashCode(), it should be used in conjunction with the division-leaving-remainder method. Because the memory address is a 32-bit integer, we only need a 31-bit non-negative integer, so the sign bit should be masked before using the division-leaving-remainder method.

```java
int hash = (x.hashCode() & 0x7fffffff) % M;
```

When using a built-in hash table implementation such as Java's HashMap, you only need to implement the hashCode() function of the Key type. Java stipulates that hashCode() can evenly distribute keys among all 32-bit integers. The hashCode() of objects such as String and Integer in Java can achieve this. The following shows how a custom type implements hashCode():

```java
public class Transaction {

    private final String who;
    private final Date when;
    private final double amount;

    public Transaction(String who, Date when, double amount) {
        this.who = who;
        this.when = when;
        this.amount = amount;
    }

    public int hashCode() {
        int hash = 17;
        int R = 31;
        hash = R * hash + who.hashCode();
        hash = R * hash + when.hashCode();
        hash = R * hash + ((Double) amount).hashCode();
        return hash;
    }
}
```

### 2. Zipper method

The zipper method uses a linked list to store keys with the same hash value to resolve conflicts.

The search requires two steps. First, search the linked list where the Key is located, and then search sequentially in the linked list.

For N keys and M linked lists (N\>M), if the hash function can satisfy the uniformity condition, the size of each linked list tends to N/M, so the number of comparisons required for miss search and insertion operations is \~N/M.

### 3. Linear detection method

The linear probing method uses gaps to resolve conflicts. When a conflict occurs, a gap is forwarded to store the conflicting key.

Using linear probing, the size of the array M should be greater than the number of keys N (M\>N).


<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/0dbc4f7d-05c9-4aae-8065-7b7ea7e9709e.gif" width="350px"> </div><br>

```java
public class LinearProbingHashST<Key, Value> implements UnorderedST<Key, Value> {

    private int N = 0;
    private int M = 16;
    private Key[] keys;
    private Value[] values;

    publicLinear
ProbingHashST() {
        init();
    }

    public LinearProbingHashST(int M) {
        this.M = M;
        init();
    }

    private void init() {
        keys = (Key[]) new Object[M];
        values = (Value[]) new Object[M];
    }

    private int hash(Key key) {
        return (key.hashCode() & 0x7fffffff) % M;
    }
}
```

##### 3.1 Search

```java
public Value get(Key key) {
    for (int i = hash(key); keys[i] != null; i = (i + 1) % M)
        if (keys[i].equals(key))
            return values[i];

    return null;
}
```

##### 3.2 Insert

```java
public void put(Key key, Value value) {
    resize();
    putInternal(key, value);
}

private void putInternal(Key key, Value value) {
    int i;
    for (i = hash(key); keys[i] != null; i = (i + 1) % M)
        if (keys[i].equals(key)) {
            values[i] = value;
            return;
        }

    keys[i] = key;
    values[i] = value;
    N++;
}
```

##### 3.3 Delete

The delete operation should reinsert all adjacent key-value pairs on the right into the hash table.

```java
public void delete(Key key) {
    int i = hash(key);
    while (keys[i] != null && !key.equals(keys[i]))
        i = (i + 1) % M;

    // Does not exist, return directly
    if (keys[i] == null)
        return;

    keys[i] = null;
    values[i] = null;

    // Re-insert the subsequent connected key-value pairs
    i = (i + 1) % M;
    while (keys[i] != null) {
        Key keyToRedo = keys[i];
        Value valToRedo = values[i];
        keys[i] = null;
        values[i] = null;
        N--;
        putInternal(keyToRedo, valToRedo);
        i = (i + 1) % M;
    }
    N--;
    resize();
}
```

##### 3.5 Adjust array size

The cost of linear detection depends on the length of consecutive entries, also called clusters. When clusters are long, many probes are also required during search and insertion. For example, the position 2\~4 in the figure below is a cluster.

α = N/M, call α the usage rate. The theory proves that when α is less than 1/2, the expected number of detections is only between 1.5 and 2.5. In order to ensure the performance of the hash table, the size of the array should be adjusted so that α is between [1/4, 1/2].

```java
private void resize() {
    if (N >= M / 2)
        resize(2 * M);
    else if (N <= M / 8)
        resize(M / 2);
}

private void resize(int cap) {
    LinearProbingHashST<Key, Value> t = new LinearProbingHashST<Key, Value>(cap);
    for (int i = 0; i < M; i++)
        if (keys[i] != null)
            t.putInternal(keys[i], values[i]);

    keys = t.keys;
    values = t.values;
    M = t.M;
}
```

## Summary

### 1. Symbol table algorithm comparison

| Algorithm | Insertion | Search | Is it in order |
| :---: | :---: | :---: | :---: |
| Unordered symbol table implemented by linked list | N | N | yes |
| Ordered symbol table implemented by binary search | N | logN | yes |
| Binary search tree | logN | logN | yes |
| 2-3 Search tree | logN | logN | yes |
| Hash table implemented by zipper method | N/M | N/M | no |
| Hash table implemented by linear detection method | 1 | 1 | no |

Hash tables should be preferred, and red-black trees should be used when ordered operations are required.

- java.util.HashMap: zipper hash table

### 3. Sparse vector multiplication

When the vector is a sparse vector, a symbol table can be used to store the non-zero indexes and values in the vector, so that the multiplication operation only needs to be performed on those non-zero elements.

```java
public class SparseVector {
    private HashMap<Integer, Double> hashMap;

    public SparseVector(double[] vector) {
        hashMap = new HashMap<>();
        for (int i = 0; i < vector.length; i++)
            if (vector[i] != 0)
                hashMap.put(i, vector[i]);
    }
public double get(int i) {
        return hashMap.getOrDefault(i, 0.0);
    }

    public double dot(SparseVector other) {
        double sum = 0;
        for (int i : hashMap.keySet())
            sum += this.get(i) * other.get(i);
        return sum;
    }
}
```
