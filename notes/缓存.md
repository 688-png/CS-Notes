## 1. Cache features

### Hit rate

When a request can be responded to by accessing the cache, it is called a cache hit.

The higher the cache hit rate, the higher the cache utilization.

### Maximum space

The cache is usually located in memory, and the memory space is usually much smaller than the disk space, so the maximum cache space cannot be very large.

When the amount of data stored in the cache exceeds the maximum space, part of the data needs to be eliminated to store newly arrived data.

### Elimination Strategy

- FIFO (First In First Out): First in, first out strategy. In real-time scenarios, where the latest data needs to be accessed frequently, FIFO can be used so that the data that enters first (the latest data) is eliminated.

- LRU (Least Recently Used): The most recently unused policy, which prioritizes the elimination of data that has not been used for the longest time, that is, the data that was last accessed the longest time ago. This strategy can ensure that the data in the memory is hot data, that is, data that is frequently accessed, thus ensuring the cache hit rate.

- LFU (Least Frequently Used): Least Frequently Used strategy, which prioritizes the elimination of data with the least usage within a period of time.

## 2. Cache location

### Browser

When the HTTP response allows caching, the browser will cache HTML, CSS, JavaScript, images and other static resources.

###ISP

The Internet Service Provider (ISP) is the first hop for network access. By caching data in the ISP, the user's access speed can be greatly improved.

### Reverse proxy

The reverse proxy is located in front of the server, and both requests and responses need to go through the reverse proxy. By caching data in the reverse proxy, the cache can be used directly to respond when the user requests the reverse proxy.

### Local cache

Use Guava Cache to cache data in the server's local memory. The server code can directly read the cache in the local memory very quickly.

### Distributed cache

Use distributed caches such as Redis and Memcache to cache data in distributed cache systems.
Compared with local cache, distributed cache is deployed separately and hardware resources can be allocated according to needs. Not only that, the server cluster can access the distributed cache, but the local cache needs to be synchronized between server clusters, which is very difficult to implement and has high performance overhead.

### Database cache

Database management systems such as MySQL have their own query caching mechanisms to improve query efficiency.

### Java internal cache

In order to optimize space and improve the creation efficiency of string and basic data type packaging classes, Java has designed a string constant pool and six packaging class buffer pools of Byte, Short, Character, Integer, Long, and Boolean.

### CPU multi-level cache

In order to solve the problem of mismatch between the computing speed and the main memory IO speed, the CPU introduces a multi-level cache structure and uses cache consistency protocols such as MESI to solve the problem of multi-core CPU cache data consistency.

## 3. CDN

Content distribution network (CDN) is an interconnected network system that uses servers closer to users to distribute static resources such as HTML, CSS, JavaScript, music, pictures, and videos to users faster and more reliably.

CDN mainly has the following advantages:

- Distribute data to users faster;
- Improve the overall bandwidth performance of the system by deploying multiple servers;
- Multiple servers can be viewed as a redundancy mechanism, resulting in high availability.

## 4. Caching problem

### Cache penetration

It refers to a request for some data that definitely does not exist. The request will penetrate the cache and reach the database.

Solution:

- Cache empty data for these non-existent data;
- Filter such requests.

### Cache Avalanche

This refers to the fact that the data is not loaded into the cache, or the cached data fails (expires) in a large area at the same time, or the cache server is down, resulting in a large number of requests reaching the database.

In a cached system, the system relies heavily on the cache, and the cache offloads a large part of the data requests. When a cache avalanche occurs, the database cannot handle such a large request, causing the database to crash.

Solution:

- In order to prevent cache avalanche caused by large-scale cache expiration at the same time, this can be achieved by observing user behavior and setting the cache expiration time reasonably;
- In order to prevent cache avalanche when the cache server goes down, distributed cache can be used. Each node in the distributed cache only caches part of the data. When a node goes down, it can ensure that the cache of other nodes is still available.
- Cache preheating can also be performed to avoid cache avalanche due to a large amount of data not being cached soon after the system is started.


### Cache consistency

Cache consistency requires that the cached data be updated in real time when the data is updated.

Solution:

- Update the cache immediately when the data is updated;
- Before reading the cache, first determine whether the cache is the latest, and if it is not the latest, update it first.

To ensure cache consistency requires a high price. It is best to cache data that does not require high consistency and allow some dirty data to exist in the cache data.

### Cache "bottomless pit" phenomenon

It refers to the phenomenon that a large number of cache nodes are added to meet business requirements, but the performance not only does not improve but also decreases.

Cause: Caching systems usually use hash functions to map keys to corresponding cache nodes. As the number of cache nodes increases, key values ​​are distributed to more nodes, causing a batch operation on the client to involve multiple network operations. This means that the time-consuming of batch operations will continue to increase as the number of nodes increases. In addition, the increase in the number of network connections will also have a certain impact on the performance of the node.

Solution:

- Optimize batch data operation commands;
- Reduce the number of network communications;
- Reduce access costs, use long connections/connection pools, NIO, etc.

## 5. Data distribution

### Hash distribution

Hash distribution is to calculate the hash value of the data and distribute it to different nodes according to the hash value. For example, if there are N nodes and the primary key of the data is key, the node number assigned to the data is: hash(key)%N.

There is a problem with the traditional hash distribution algorithm: when the number of nodes changes, that is, the value of N changes, then almost all the data needs to be redistributed, which will lead to a large amount of data migration.

### Sequential distribution

Divide the data into multiple consecutive parts and distribute them to different nodes according to the data ID or time. For example, the ID range of the User table is 1 \~ 7000. It can be divided into multiple sub-tables using sequential distribution. The corresponding primary key range is 1 \~ 1000, 1001 \~ 2000,..., 6001 \~ 7000.

The main advantages of sequential distribution compared to hash distribution are as follows:

- Ability to maintain the original order of data;
- And can accurately control the amount of data stored on each server, thereby maximizing storage space utilization.

### Basic principles

Think of the hash space [0, 2<sup>n</sup>-1] as a hash ring, and each server node is configured on the hash ring. After each data object obtains a hash value through hash modulo, it is stored in the first node in the clockwise direction of the hash ring that is greater than or equal to the hash value.

### Virtual node

The consistent hashing described above has the problem of uneven data distribution, and the amount of data stored by nodes may be very different.

Data unevenness is mainly due to the uneven distribution of nodes on the hash ring. This situation is especially obvious when the number of nodes is small.

The solution is to add virtual nodes and then map the virtual nodes to real nodes. If the number of virtual nodes is much greater than that of real nodes, the uniformity of the distribution of virtual nodes on the hash ring will be better than that of the original real nodes, thus making the data distribution more even.

## 7. LRU

The following is the implementation of the LRU algorithm based on doubly linked list + HashMap. The algorithm is explained as follows:

- When a node is accessed, delete it from its original position and reinsert it at the head of the linked list. This ensures that the most recently unused nodes are stored at the end of the linked list. When the number of nodes exceeds the maximum cache space, the nodes at the end of the linked list will be eliminated.
- In order to make the time complexity of the deletion operation O(1), a node cannot be found by traversal. HashMap stores the mapping from Key to nodes. Through the Key, the node can be obtained in O(1) time, and then deleted from the two-way queue in O(1) time.

            this.k = k;

    public LRU(int maxSize) {

        this.maxSize = maxSize;
        this.map = new HashMap<>(maxSize * 4 / 3);

        head = new Node(null, null);
        tail = new Node(null, null);

        head.next = tail;
        tail.pre = head;

    public V get(K key) {

        if (!map.containsKey(key)) {
            return null;
        }

        Node node = map.get(key);
        unlink(node);
        appendHead(node);

        return node.v;
    }

        pre.next = next;

        node.pre = null;

        node.next = next;

        tail.pre = pre;
        pre.next = tail;

        node.pre = null;
