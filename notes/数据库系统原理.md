## 1. Affairs

### Concept

A transaction refers to a set of operations that meet ACID characteristics. You can commit a transaction through Commit or rollback using Rollback.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191207222237925.png"/> </div><br>

### ACID

#### 1. Atomicity

Transactions are regarded as the smallest indivisible unit. All operations of a transaction are either submitted successfully or rolled back after failure.

Rollback can be implemented using the rollback log (Undo Log). The rollback log records the modification operations performed by the transaction. These modification operations can be performed in reverse during rollback.

#### 2. Consistency

The database maintains a consistent state before and after a transaction is executed. In the consistency state, the results of all transactions reading the same data are the s
ame.

#### 3. Isolation

Modifications made by one transaction are not visible to other transactions until they are finally committed.

#### 4. Durability

Once a transaction is committed, the changes made will be permanently saved in the database. Even if the system crashes, the results of transaction execution cannot be lost.

If the system crashes, the redo log can be used to recover, thereby achieving durability. Unlike rollback logging that records logical modifications to data, redo logging records physical modifications to data pages.

----

The ACID characteristics of transactions are simple in concept, but not easy to understand, mainly because these characteristics are not in a horizontal relationship:

- Only when consistency is met, the execution result of the transaction will be correct.
- In the absence of concurrency, transactions are executed serially and isolation must be met. At this time, as long as atomicity can be satisfied, consistency will definitely be satisfied.
- In the case of concurrency, multiple transactions are executed in parallel. Transactions must not only meet atomicity, but also isolation to meet consistency.
- Transactions satisfy persistence in order to cope with system crashes.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191207210437023.png"/> </div><br>

###AUTOCOMMIT

MySQL uses autocommit mode by default. That is, if you do not explicitly use the `START TRANSACTION` statement to start a transaction, then each query operation will be treated as a transaction and automatically committed.

## 2. Concurrency consistency issues

In a concurrent environment, it is difficult to ensure the isolation of transactions, so many concurrency consistency problems will occur.

### Lost modifications

Lost modification means that the update operation of one transaction is replaced by the update operation of another transaction. It is often encountered in real life. For example, two transactions, T<sub>1</sub> and T<sub>2</sub>, both modify the same data. T<sub>1</sub> is modified first and submitted to take effect. T<sub>2</sub> is modified subsequently. The modification of T<sub>2</sub> covers the modification of T<sub>1</sub>.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191207221744244.png"/> </div><br>

### Read dirty data

Reading dirty data means that under different transactions, the current transaction can read uncommitted data from another transaction. For example: T<sub>1</sub> modifies a piece of data but does not commit it, and T<sub>2</sub> then reads the data. If T<sub>1</sub> undoes this modification, then the data read by T<sub>2</sub> is dirty data.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191207221920368.png"/> </div><br>

### Non-repeatable read

Non-repeatable read refers to reading the same data set multiple times within a transaction. Befo
re this transaction ended, another transaction also accessed the same
The data set is merged and modified. Due to the modification of the second transaction, the data read twice in the first transaction may be inconsistent. For example: T<sub>2</sub> reads a piece of data, and T<sub>1</sub> modifies the data. If T<sub>2</sub> reads this data again, the result read at this time is different from the result read the first time.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191207222102010.png"/> </div><br>

### Phantom Reading

Phantom reading is essentially a case of non-repeatable reading. T<sub>1</sub> reads a certain range of data, T<sub>2</sub> inserts new data in this range, and T<sub>1</sub> reads this range of data again. The result of the read is different from the result of the first read.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191207222134306.png"/> </div><br>

----

The main reason for concurrency inconsistency is the destruction of transaction isolation. The solution is to ensure isolation through concurrency control. Concurrency control can be achieved through blocking, but the blocking operation requires user control and is quite complicated. The database management system provides transaction isolation levels to allow users to deal with concurrency consistency issues in an easier way.

## 3. Blockade

### Blocking granularity

MySQL provides two locking granularities: row-level locks and table-level locks.

You should try to lock only the part of the data that needs to be modified, rather than all resources. The smaller the amount of data locked, the smaller the possibility of lock contention and the higher the degree of concurrency of the system.

However, locking requires resources, and various lock operations (including acquiring locks, releasing locks, and checking lock status) will increase system overhead. Therefore, the smaller the block granularity, the greater the system overhead.

When choosing a locking granularity, a trade-off needs to be made between lock overhead and concurrency.


### Blocking type

#### 1. Read-write lock

- Mutex lock (Exclusive), abbreviated as X lock, also known as write lock.
- Shared lock (Shared), abbreviated as S lock, also known as read lock.

There are two provisions:

- A transaction adds an X lock to data object A, and can read and update A. During the locking period, other transactions cannot add any locks to A.
- A transaction adds an S lock to data object A, and can perform read operations on A, but cannot perform update operations. During the locking period, other transactions can add S lock to A, but cannot add X lock.

Intention lock introduces IX/IS on top of the original X/S lock. IX/IS are table locks and are used to indicate that a transaction wants to add an X lock or S lock to a certain data row in the table. There are two provisions:

### Blocking Protocol

#### 1. Level 3 Blockade Agreement

**Level 1 Blockade Agreement**

When transaction T wants to modify data A, it must acquire X lock, and the lock will not be released until T ends.

The problem of lost modifications can be solved because two transactions cannot modify the same data at the same time, so the modifications of the transaction will not be overwritten.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191207220440451.png"/> </div><br>

**Level 2 Lockdown Protocol**

On the basis of one level, it is required that the S lock must be added when reading data A, and the S lock must be released immediately after reading.

It can solve the problem of reading dirty data, because if a transaction modifies data A, according to the level 1 blocking protocol, an X lock will be added, and then an S lock cannot be added, that is, the data will not be read.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191207220831843.png"/> </div><br>

**Level Three Lockdown Agreement**

On the basis of the second level, it is required that the S lock must be added when reading data A, and the S lock cannot be released until the end of the transaction.

It can solve the problem of non-repeatable reading, because when reading A, other trans
actions cannot add X lock to A, thus avoiding data changes during reading.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191207221313819.png"/> </div><br>

#### 2. Two-stage lock protocol

Locking and unlocking are performed in two stages.

Serializable scheduling means that through concurrency control, the results of concurrently executed transactions are the same as the results of a serially executed transaction. Transactions executed serially do not interfere with each other, and there will be no concurrency consistency issues.

Transactions following the two-stage lock protocol are sufficient conditions to ensure serializable scheduling. For example, the following operation satisfies the two-stage lock protocol, which is serializable scheduling.

```html
lock-x(A)...lock-s(B)..
.lock-s(C)...unlock(A)...unlock(C)...unlock(B)
```

But it is not a necessary condition. For example, the following operation does not satisfy the two-stage lock protocol, but it can still be scheduled serially.

```html
lock-x(A)...unlock(A)...lock-s(B)...unlock(B)...lock-s(C)...unlock(C)
```

### MySQL implicit and explicit locking

MySQL's InnoDB storage engine uses a two-stage locking protocol, which automatically locks when needed based on the isolation level, and all locks are released at the same time. This is called implicit locking.

InnoDB can also use specific statements for explicit locking:

```sql
SELECT ... LOCK IN SHARE MODE;
SELECT ... FOR UPDATE;
```

## 4. Isolation level

### READ UNCOMMITTED

Modifications in a transaction are visible to other transactions even if they are not committed.

### READ COMMITTED

A transaction can only read modifications made by committed transactions. In other words, changes made by one transaction are not visible to other transactions until they are committed.

### REPEATABLE READ

It is guaranteed that the result of reading the same data multiple times in the same transaction is the same.

### SERIALIZABLE

Force transactions to be executed serially, so that multiple transactions do not interfere with each other and there will be no concurrent consistency issues.

This isolation level requires locking, because the locking mechanism is used to ensure that only one transaction is executed at the same time, that is, to ensure that transactions are executed serially.

## 5. Multi-version concurrency control

Multi-Version Concurrency Control (MVCC) is a specific way for MySQL's InnoDB storage engine to implement isolation levels. It is used to implement two isolation levels: committed read and repeatable read. The uncommitted read isolation level always reads the latest data row, which has very low requirements and does not require the use of MVCC. The serializable isolation level requires locking all read rows, which cannot be achiev
ed using MVCC alone.

### Basic idea

As mentioned in the blocking section, locking can solve the concurrency consistency problem that occurs when multiple transactions are executed at the same time. In actual scenarios, there are often more reading operations than writing operations, so read-write locks are introduced to avoid unnecessary locking operations. For example, there is no mutual exclusion relationship between reading and reading. Read and write operations in read-write locks are still mutually exclusive, and MVCC uses the idea of ​​multi-version. The write operation updates the latest version snapshot, while the read operation reads the old version snapshot. There is no mutual exclusion relationship. This is similar to CopyOnWrite.

In MVCC, transaction modification operations (DELETE, INSERT, UPDATE) will create a new version snapshot for the data row.

The most fundamental reason for dirty reads and non-repeatable reads is that transactions read uncommitted modifications from other transactions. When a transaction performs a read operation, in order to solve the problems of dirty reads and non-repeatable reads, MVCC stipulates that only committed snapshots can be read. Of course, a transaction can read its own uncommitted snapshot, which is not considered a dirty read.

### Version number

- System version number SYS_ID: It is an increasing number. Every time a new transaction is started, the system version number will automatically increase.
- Transaction version number TRX_ID: The system version number when the transaction starts.

### Undo Log

Multi-version of MVCC refers to multiple versions of snapshots, which are stored in the Undo log, which connects all snapshots of a data row through the rollback pointer ROLL_PTR.

For example, create a table t in MySQL, containing the primary key id and a field x. We first insert a data row and then perform two update operations on the data row.

```sql
INSERT INTO t(id, x) VALUES(1, "a");
UPDATE t SET x="b" WHERE id=1;
UPDATE t SET x="c" WHERE id=1;
```

Because `START TRANSACTION` is not used to execute the above operation as a transaction, according to MySQL's AUTOCOMMIT mechanism, each operation will be executed as a transaction, so the above operation involves a total of three transactions. In addition to recording the transaction version number TRX_ID and operation, the snapshot also records a bit DEL field to mark whether it is deleted.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191208164808217.png"/> </div><br>

INSERT, UPDATE, and DELETE operations create a log and write the transaction version number TRX_ID. DELETE can be thought of as a special UPDATE that additionally sets the DEL field to 1.

### ReadView

MVCC maintains a ReadView structure, which mainly contains the current system's uncommitted transaction list TRX_IDs {TRX_ID_1, TRX_ID_2, ...}, as well as the minimum values TRX_ID_MIN and TRX_ID_MAX of the list.

<div
align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191208171445674.png"/> </div><br>

When performing a SELECT operation, determine whether the data row snapshot can be used based on the relationship between the TRX_ID of the data row snapshot, TRX_ID_MIN and TRX_ID_MAX:

- TRX_ID \< TRX_ID_MIN, indicating that the data row snapshot was changed before all current uncommitted transactions, so it can be used.

- TRX_ID \> TRX_ID_MAX, indicating that the data row snapshot was changed after the transaction was started and therefore cannot be used.
- TRX_ID_MIN \<= TRX_ID \<= TRX_ID_MAX, it needs to be judged according to the isolation level:
  - Read commit: If TRX_ID is in the TRX_IDs list, it means that the transaction corresponding to the data row snapshot has not yet been committed, and the snapshot cannot be used. Otherwise, it means it has been submitted and can be used.
  - Repeatable Read: Neither can be used. Because if it can be used, other transactions can also read this data row snapshot and modify it. Then the value obtained by the current transaction when reading this data row will change, which means that a non-repeatable read problem occurs.

When the data row snapshot is unavailable, you need to find the next snapshot along the rollback pointer ROLL_PTR of the Undo Log, and then make the above judgment.

### Snapshot read and current read

#### 1. Snapshot reading

The SELECT operation of MVCC is the data in the snapshot and does not require locking operations.

```sql
SELECT * FROM table ...;
```

#### 2. Current reading

MVCC other operations that modify the database (INSERT, UPDATE, DELETE) require locking operations to read the latest data. It can be seen that MVCC does not eliminate the need for locking at all, but only avoids SELECT
locking operation.

```sql
INSERT;
UPDATE;
DELETE;
```

When performing a SELECT operation, you can force a locking operation to be specified. The first statement below requires an S lock, and the second statement requires an X lock.

```sql
SELECT * FROM table WHERE ? lock in share mode;
SELECT * FROM table WHERE ? for update;
```

## 6. Next-Key Locks

Next-Key Locks are a lock implementation of MySQL's InnoDB storage engine.

MVCC cannot solve the phantom read problem, and Next-Key Locks exist to solve this problem. Under the repeatable read (REPEATABLE READ) isolation level, use MVCC + Next-Key Locks to solve the phantom read problem.

### Record Locks

Locks the index on a record, not the record itself.

If the table is not indexed, InnoDB will automatically create a hidden clustered index on the primary key, so Record Locks can still be used.

### Gap Locks

Locks the gaps between indexes, but not the indexes themselves. For example, when a transaction executes the following statement, other transactions cannot insert 15 in t.c.

```sql
SELECT c FROM t WHERE c BETWEEN 10 and 20 FOR UPDATE;
```

### Next-Key Locks

It is a combination of Record Locks and
Gap Locks, which not only locks the index on a record, but also locks the gaps between indexes. It locks an open and closed range. For example, if an index contains the following values: 10, 11, 13, and 20, then the following range needs to be locked:

```sql
(-∞, 10]
(10, 11]
(11, 13]
(13, 20]
(20, +∞)
```

## 7. Relational database design theory

### Functional dependencies

Notation A-\>B means that function A determines function B. It can also be said that function B depends on function A.

If {A1, A2, ..., An} is a set of one or more attributes of the relationship, and the set function determines all other attributes of the relationship and is the smallest, then the set is called a key code.

For A-\>B, if the proper subset A' of A can be found such that A'-\>B, then A-\>B is a partial functional dependence, otherwise it is a complete functional dependence.

For A-\>B, B-\>C, then A-\>C is a transitive functional dependency.

### Exception

The functional dependency of the following student course relationship is {Sno, Cname} -\> {Sname, Sdept, Mname, Grade}, and the key code is {Sno, Cname}. In other words, after identifying students and courses, other information can be determined.

| Sno | Sname | Sdept | Mname | Cname | Grade |
| :---: | :---: | :---: | :---: | :---: |:---:|
| 1 | Student-1 | College-1 | Dean-1 | Course-1 | 90 |
| 2 | Student-2 | College-2 | Dean-2 | Course-2 | 80 |
| 2 | Student-2 | College-2 | Dean-2 | Course-1 | 100 |
| 3 | Student-3 | College-2 | Dean-2 | Course-2 | 95 |

Relationships that do not conform to the paradigm will produce many exceptions, mainly including the following four exceptions:

- Redundant data: e.g. `Student-2` appears twice.
- Modification exception: The information in one record is modified, but the same information in another record is not modified.
- Deletion exception: If one piece of information is deleted, other information will also be lost. For example, if you delete `Course-1`, you need to delete the first and third rows, then the information of `Student-1` will be lost.
- Insertion exception: For example, if you want to insert a student's information, if the student has not chosen a course, you cannot insert it.

### Paradigm

Paradigm theory is to solve the four anomalies mentioned above.

High-level normal forms depend on low-level normal forms, and 1NF is the lowest level normal form.

#### 1. First Normal Form (1NF)

Properties are inseparable.

#### 2. Second Normal Form (2NF)

Each non-primary attribute is fully functionally dependent on the key code.

Can be satisfied by decomposition.
