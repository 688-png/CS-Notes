## 1. Load balancing

Application servers (nodes) in a cluster are usually designed to be stateless, and users can request any node.

The load balancer will forward user requests to the appropriate node based on the load of each node in the cluster.

Load balancers can be used to achieve high availability and scalability:

- High availability: When a node fails, the load balancer will forward user requests to another node to ensure that all services continue to be available;
- Scalability: Nodes can be easily added or removed depending on the overall system load.

The load balancer operation process consists of two parts:

1. Obtain the forwarding node according to the load balancing algorithm;
2. Forward.

### Load balancing algorithm

#### 1. Round Robin

The polling algorithm sends each request to each server in turn.

In the figure below, a total of 6 clients generated 6 requests, and these 6 requests were sent in the order of (1, 2, 3, 4, 5, 6). Requests for (1, 3, 5) will be sent to server 1, and requests for (2, 4, 6) will be sent to server 2.

This algorithm is more suitable for scenarios where the performance of each server is similar. If there are differences in performance, the server with poor performance may not be able to bear the excessive load (Server 2 in the figure below).

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/bfea8772-d01b-4a51-8adc-edfd7d3dce84.jpg" width="500px"/> </div><br>

#### 2. Weighted Round Robbin

Weighted polling is based on polling and assigns certain weights to servers based on their performance differences. Servers with high performance are assigned higher weights.

For example, in the figure below, server 1 is assigned a weight of 5 and server 2 is assigned a weight of 1. Then the (1, 2, 3, 4, 5) request will be sent to server 1, and the (6) request will be sent to server 2.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/4ab87717-e264-4232-825d-8aaf08f14e8b.jpg" width="500px"/> </div><br>

#### 3. Least Connections

Since the connection time of each request is different, using polling or weighted polling algorithm may make the current number of connections of one server too large, while the number of connections of another server is too small, resulting in load imbalance.

For example, in the figure below, requests (1, 3, 5) are sent to server 1, but (1, 3) is quickly disconnected, and only (5) requests to connect to server 1; (2, 4, 6) requests are sent to server 2, and only (2)'s connection is disconnected, and (6, 4) requests to connect to server 2. While the system continues to operate, Server 2 will be under an excessive load.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/e98deb5a-d5d4-4294-aa9b-9220d4483403.jpg" width="500px"/> </div><br>

The least connection algorithm is to send the request to the server with the least current number of connections.

For example, in the figure below, server 1 currently has the smallest number of connections, so the newly arrived request 6 will be sent to server 1.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/43d323ac-9f07-4e4a-a315-4eaf8c38766c.jpg" width="500px"/> </div><br>

#### 4. Weighted Least Connection

On the basis of the minimum number of connections, each server is assigned a weight according to its performance, and then the number of connections each server can handle is calculated based on the weight.

#### 5. Random algorithm (Random)

Send requests to the server randomly.

Similar to the polling algorithm, this algorithm is more suitable for scenarios where server performance is about the same.

#### 6. Source
Address Hash (IP Hash)

The source address hash calculates the hash value of the client IP and then modulo the number of servers to obtain the sequence number of the target server.

It can ensure that requests from clients with the same IP will be forwarded to the same server to achieve sticky session.

### Forwarding implementation

#### 1. HTTP Redirect

After the HTTP redirection load balancing server uses a certain load balancing algorithm to calculate the server's IP address, it writes the address into the HTTP redirection message with a status code of 302. After receiving the redirect message, the client needs to re-initiate a request to the server.

Disadvantages:

- Two requests are required, so the access delay is relatively high;
- The HTTP load balancer has limited processing power, which limits the size of the cluster.

The disadvantages of load balancing forwarding are obvious, and it is rarely used in actual scenarios.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/02a1fbfd-7a9d-4114-95df-ca2445587a1f.jpg" width="500px"/> </div><br>

#### 2. DNS domain name resolution

A load balancing algorithm is used to calculate the server IP address while DNS resolves the domain name.

Advantages:

- DNS can perform domain name resolution based on geographical location and return the IP address of the server closest to the user.

Disadvantages:

- Since DNS has a multi-level structure, domain name records at each level may be cached. When an offline server needs to modify DNS records, it will take a long time to take effect.

Large websites basically use DNS as the first-level load balancing method, and then use other methods internally for second-level load balancing. In other words, the result of domain name resolution is the internal load balancing server IP address.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/d2c072cc-8b17-480c-813e-18cdb3b4b51f.jpg" width="500px"/> </div><br>

#### 3. Reverse proxy server

The reverse proxy server is located in front of the origin server, and the user's request needs to go through the reverse proxy server before reaching the origin server. Reverse proxy can be used for caching, logging, etc., and can also be used as a load balancing server.

In this load balancing forwarding method, the client does not directly request the source server, so the source server does not need an external IP address, and the reverse proxy needs to be configured with two sets of internal and external IP addresses.

Advantages:

- Integrated with other functions, easy to deploy.

Disadvantages:

- All requests and responses need to go through the reverse proxy server, which can become a performance bottleneck.

#### 4. Network layer

Obtain network data packets in the operating system kernel process, calculate the IP address of the source server according to the load balancing algorithm, modify the destination IP address of the request data packet, and finally forward it.

The response returned by the source server also needs to go through the load balancing server, which is usually implemented by having the load balancing server also serve as the gateway server of the cluster.

Advantages:

- Processed in the kernel process, the performance is relatively high.

Disadvantages:

- Like a reverse proxy, all requests and responses pass through the load balancing server, which can become a performance bottleneck.

#### 5. Link layer

At the link layer, the MAC address of the source server is calculated based on the load balancing algorithm, and the destination MAC address of the request packet is modified and forwarded.

By configuring the virtual IP address of the source server to be consistent with the IP address of the load balancing server, forwarding can be performed without modifying the IP address. Precisely because the IP addresses are the same, the response from the source server does not need to be forwarded back to the load balancing server, but can be forwarded directly to the client, avoiding the load balancing server from becoming a bottleneck.

This is a delta transmission mode known as direct routing. For websites that provide download and video services, direct routing avoids a large amount of network transmission data passing through the load balancing server.

This is currently the most widely used load balancing forwarding method for large websites. The load balancing server that can be used on the Linux platform is LVS (Linux Virtual Server).

## 2. Session management under the cluster

If a user's Session information is stored on a server, then when the load balancer forwards the user's next request to another server, since the server does not have the user's Session information, the user will need to log in again.

### Sticky Session

You need to configure the load balancer so that all requests from a user are routed to the same server, so that the user's Session can be stored in that server.

Disadvantages:

- When the server goes down, all Sessions on the server will be lost.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/1623f524-b011-40c8-b83f-eab38d538f76.png" width="400px"/> </div><br>

### Session Replication

Session synchronization is performed between servers. Each server has Session information of all users, so users can make requests to any server.

Disadvantages:

- Taking up too much memory;
- The synchronization process consumes network bandwidth and server processor time.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/40c6570d-c1d7-4c38-843e-ba991b2328c2.png" width="400px"/> </div><br>

### Session Server

Use a separate server to store Session data, you can use traditional MySQL, or use in-memory databases such as Redis or Memcached.

Advantages:

- In order to make large websites scalable, application servers in a cluster usually need to remain stateless, so the application server cannot store user session information. Session Server stores user session information separately, thus ensuring the statelessness of the application server.

Disadvantages:

- Need to implement code to access Session.
