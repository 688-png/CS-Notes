##Decorator

### Intent

Dynamically add functionality to objects.

### Class Diagram

Both Decorator and ConcreteComponent inherit from Component. The method implementation of a concrete component does not need to depend on other objects, and a decorator combines a component so that it can decorate other decorators or concrete components. The so-called decoration is to put the decorator on the decorated object, thereby dynamically expanding the function of the decorated object. Part of the decorator's method is its own, which belongs to its function, and then it is implemented by calling the decorator's method, thus retaining the function of the decorator. It can be seen that the specific component should be the lowest level of the decoration hierarchy, because only the method implementation of the specific component does not need to depend on other objects.

### Implementation

Design different types of drinks. Ingredients can be added to the drinks, such as milk, and new ingredients can be added dynamically. Calculate the price of a drink as each additional ingredient increases the price of the drink.

The image below shows the addition of the new Mocha topping on the DarkRoast drink, followed by the addition of the Whip topping. DarkRoast is wrapped in Mocha, which is wrapped in Whip. They all inherit from the same parent class and have a cost() method. The cost() method of the outer class calls the cost() method of the inner class.

```java
public interface Beverage {
    double cost();
}
```

```java
public class DarkRoast implements Beverage {
    @Override
    public double cost() {
        return 1;
    }
}
```

```java
public class HouseBlend implements Beverage {
    @Override
    public double cost() {
        return 1;
    }
}
```

```java
public abstract class CondimentDecorator implements Beverage {
    protected Beverage beverage;
}
```

```java
public class Milk extends CondimentDecorator {

    public Milk(Beverage beverage) {
        this.beverage = beverage;
    }

    @Override
    public double cost() {
        return 1 + beverage.cost();
    }
}
```

```java
public class Mocha extends CondimentDecorator {

    public Mocha(Beverage beverage) {
        this.beverage = beverage;
    }

    @Override
    public double cost() {
        return 1 + beverage.cost();
    }
}
```

```java
public class Client {

    public static void main(String[] args) {
        Beverage beverage = new HouseBlend();
        beverage = new Mocha(beverage);
        beverage = new Milk(beverage);
        System.out.println(beverage.cost());
    }
}
```

### Design principles

Classes should be open for extension and closed for modificatio
n: that is, adding new functionality does not require modifying the code. Beverages can dynamically add new ingredients without having to modify the drink code.

It is impossible to design all classes to meet this principle, and this principle should be applied where changes are most likely to occur.
