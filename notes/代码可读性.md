# 1. The importance of readability

A large part of programming is reading code, not only reading your own code, but also reading other people's code. Therefore, code with good readability can greatly improve programming efficiency.

Code with good readability often leads to better code structure, because programmers are more willing to modify this part of the code, and it is easier to modify.

Readability can only be given up for efficiency in core areas, otherwise readability comes first.

# 2. Use names to express the meaning of the code

Some more expressive words:

| word | alternative word |
| :---: | --- |
| send | deliver, dispatch, announce, distribute, route |
| find | search, extract, locate, recover |
| start| launch, create, begin, open|
| make | create, set up, build, generate, compose, add, new |

Using i, j, k as the names of loop iterators is too simple. Names such as user_i and member_i would be more expressive. Because the more loop levels there are, the more difficult the code is to understand, and expressive iterator names will be more readable.

Adding information such as adjectives to a name can make the name more expressive, but it can also make the name longer. The rule of thumb for name length is: the larger the scope, the longer the name. Therefore some simple names can only be used in short scopes.

# 3. The name cannot bring ambiguity

After choosing a name, you should think about how others will interpret the name and whether they will misunderstand the original meaning.

Boolean related names are prefixed with is, can, should, has, etc.

- Use min and max to represent the quantity range;
- Use first and last to indicate the inclusive range of the access space;

- begin and end represent the excluded range of the access space, that is, end does not include the tail.

# 5. Why write comments?

When reading the code, you will first notice the comments. If the comments are not of much use, then the time of reading the code will be wasted. Code whose meaning can be directly seen does not need to be commented, especially those that do not need to be commented for each method, such as those simple getter and setter methods. Writing comments for these methods will make the code less readable.

Don't just give it a random name just because there are comments, but strive to have a good name without writing comments.

Comments can be used to document the thought process leading to the current solution, making it easier for readers to understand the code.

Comments are used to remind you of special situations.

Mark it with TODO etc.:

| Tags | Usage |
|---|---|
|TODO| To be done |
|FIXME| To be fixed |
|HACK| Rough solution |
|XXX| Danger! Here are the important questions |

# 7. Improve the readability of control flow

In a conditional expression, the left side is a variable and the right side is a constant. For example, the first statement below is correct:

```java
if (len < 10)
if (10 > len)
```

Only use the ? : ternary operator to make the code more compact when the logic is simple, otherwise it should be split into if / else;

The conditions of do / while are placed at the end, which is not simple and clear enough, and there will be some confusion. It is best to use while instead.

If there is only one goto target, then goto is still acceptable, but an overly complex goto will make the code very unreadable, so goto should be avoided.

In nested loops, using some return statements can often reduce the number of nesting levels.

# 9. Variables and readability

**Remove control flow variables**. You can redu
ce the use of control flow variables by using break or return in a loop.

```java
boolean done = false;
while (/* condition */ && !done) {
    ...
    if (...) {
        done = true;
        continue;
    }
}
```

```java
while(/* condition */) {
    ...
    if (...) {
        break;
    }
}
```
**Reduce variable scope**. The smaller the scope, the easier it is to locate all places where the variable is used.

JavaScript can use closures to reduce scope. In the following code, submit_form is a function variable, and the submitted variable controls that the function will not be submitted twice. In the first implementation, submit is a global variable, and in the second implementation, submit is placed in an anonymous function, thus limiting the scope of scope.

```js
submitted = false;
var submit_form = function(form_name) {
    if (submitted) {
        return;
    }
    submitted = true;
};
```

```js
var submit_form = (function() {
    var submitted = false;
    return function(form_name) {
        if(submitted) {
            return;
        }
        submitted = true;
    }
}()); // () causes the outer anonymous function to execute immediately
```

Variables in JavaScript that are not declared with var are global variables, and global variables can be confusing, so you should always use var to declare variables.

A variable should be defined closest to where it is used.

# 10. Extraction function

Engineering is about breaking big problems into smaller ones and putting the solutions to those problems back together.

First, you should clarify the high-level goal of a function, and then extract the code that does not directly work for this goal and put it into an independent function.

Introductory c
ode:

```java
int findClostElement(int[] arr) {
    int clostIdx;
    int clostDist = Interger.MAX_VALUE;
    for (int i = 0; i < arr.length; i++) {
        int x = ...;
        int y = ...;
        int z = ...;
        int value = x * y * z;
        int dist = Math.sqrt(Math.pow(value, 2), Math.pow(arr[i], 2));
        if (dist < clostDist) {
            clostIdx = i;
            clostDist = value;
        }
    }
    return clostIdx;
}
```

The loop part in the above code mainly calculates distance. This part does not belong to the high-level goal of the code. The high-level goal is to find the value of the minimum distance, so this part can be extracted into an independent function instead. This also brings additional benefits: it can be tested independently, and program errors can be quickly found and modified.


It’s not that the more functions you extract, the better. If you extract too much, you may need to keep jumping around when reading the code. Only when the current function can express its content without knowing the details of a certain piece of code, is it good to extract this piece of code into a sub-function.

Function extraction is also used to reduce code redundancy.
