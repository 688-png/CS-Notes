## 1. Distributed lock

In a stand-alone scenario, you can use the language's built-in lock to achieve process synchronization. However, in a distributed scenario, the processes that need to be synchronized may be located on different nodes, so distributed locks need to be used.

Blocking locks are usually implemented using mutexes:

- A mutex of 0 means that other processes are using the lock, and it is in a locked state at this time;
- A mutex of 1 indicates an unlocked state.

1 and 0 can be represented by an integer value, or by whether a certain data exists.

### Unique index of database

A record is inserted into the table when the lock is acquired and deleted when the lock is released. A unique index ensures that the record is only inserted once
, then you can use whether this record exists to determine whether it is in a locked state.

There are several problems:

- The lock has no expiration time. If the unlocking fails, other processes will no longer be able to obtain the lock;
- It can only be a non-blocking lock. If the insertion fails, an error will be reported directly and it cannot be retried;
- Not reentrant, the process that has already acquired the lock must also acquire the lock again.

### Redis SETNX command

Use the SETNX (set if not exist) instruction to insert a key-value pair. If the Key already exists, False will be returned. Otherwise, the insertion is successful and True will be returned.

The SETNX instruction is similar to the unique index of the database, ensuring that there is only one key-value pair of Key. Then you can use the existence of a Key's key-value pair to determine whether it is in a locked state.

The EXPIRE directive can set an expiration time for a key-value pair, thereby avoiding the problem of lock release failure in the database's unique index implementation.

### RedLock algorithm of Redis

Multiple Redis instances are used to implement distributed locks to ensure that they are still available in the event of a single point of failure.

- Try to acquire locks from N independent Redis instances;
- Calculate the time it takes to acquire the lock. Only when the time is less than the expiration time of the lock and the lock is acquired from most (N / 2 + 1) instances, the lock acquisition is considered successful;
- If acquiring the lock fails, release the lock on each instance.

### Zookeeper’s ordered nodes

#### 1. Zookeeper abstract model

Zookeeper provides a tree-structured namespace, and the parent node of the /app1/p_1 node is /app1.

#### 2. Node type

- Permanent nodes: will not disappear due to session end or timeout;
- Temporary node: will disappear if the session ends or times out;
- Ordered node: A numeric suffix will be added to the end of the node name, and it is in order. For example, the generated ordered node is /lock/node-0000000000, and its next ordered node is /lock/node-0000000001, and so on.

#### 3. Listener

Register a listener for a node and send a message to the client when the node status changes.

#### 4. points
Cloth lock implementation

- Create a lock directory /lock;
- When a client needs to acquire a lock, create temporary and ordered child nodes under /lock;
- The client obtains the child node list under /lock and determines whether the child node it created is the child node with the smallest sequence number in the current child node list. If so, it is considered to have obtained the lock; otherwise, it monitors its previous child node, obtains the change notification of the child node, and repeats this step until the lock is obtained;
- Execute the business code and delete the corresponding child node after completion.

#### 5. Session timeout

If a session that has obtained the lock times out, because a temporary node is created, the temporary node corresponding to the session will be deleted, and other sessions can obtain the lock. It can be seen that this implementation method will not cause the problem of the database's unique index implementation method failing to release the lock.

#### 6. Herding effect

If a node does not obtain the lock, it only needs to monitor its previous child node. This is because if all child nodes are monitored, then all other child nodes will receive notifications if the status of any child node changes (herding effect, when one sheep moves, other sheep will flock to it), and we only want its subsequent child node to receive notifications.

## 2. Distributed transactions

It means that the transaction operations are located on different nodes, and the ACID characteristics of the transaction need to be guaranteed.

For example, in an order scenario, if inventory and orders are not on the same node, distributed transactions are involved.

The difference between distributed locks and distributed transactions:

- The key to the lock problem lies in the mutual exclusion relationship between process operations. For example, multiple processes modify the balance of an account at the same time. If there is no mutual exclusion relationship, the balance of the account will be incorrect.
- The key to the transaction problem is that a series of operations involved in the transaction need to meet the ACID characteristics. For example, to satisfy atomic operations, these operations need to be executed either all or none.

### 2PC

Two-phase Commit (2PC) introduces a coordinator to coordinate the behavior of participants and ultimately decides whether these participants actually want to execute the transaction.

#### 1. Running process

##### 1.1 Preparation stage

The coordinator asks the participants whether the transaction was executed successfully, and the participants send back the transaction execution results. A query can be thought of as a vote that requires the consent of all participants before it can be executed.

#### 2. Problems

##### 2.1 Synchronous blocking

All transaction participants are in a synchronous blocking waiting state while waiting for responses from other participants and cannot perform other operations.

##### 2.2 Single point problem

The coordinator plays a very important role in 2PC, and failure will have a great impact. Especially if a failure occurs during the submission phase, all participants will be blocked and waiting synchronously, unable to complete other operations.

##### 2.3 Data inconsistency

In the commit phase, if the coordinator only sends part of the Commit message and an exception occurs on the network at this time, only some participants receive the Commit message, which means that only some participants commit the transaction, making the system data inconsistent.

##### 2.4 Too conservative

The failure of any node will cause the entire transaction to fail, and there is no perfect fault tolerance mechanism.

### Local message table

The local message table and the business data table are in the same database, so local transactions can be used to ensure that operations on the two tables meet transaction characteristics, and message queues are used to ensure eventual consistency.

1. After one party of the distributed transaction operation completes the operation of writing business data, it sends a message to the local message table. The local transaction can guarantee that the message will be written to the local message table.
2. Then forward the message in the local message table to the message queue. If the forwarding is successful, the message will be deleted from the local message table, otherwise it will continue to be re-forwarded.
3. Read a message from the message queue on the other side of the distributed transaction operation and perform the operations in the message.

## 3. CAP

It is impossible for a distributed system to satisfy all requirements at the same time
Consistency (C: Consistency), availability (A: Availability) and partition tolerance (P: Partition Tolerance), only two of them can be satisfied at the same time.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/a14268b3-b937-4ffa-a34a-4cc53071686b.jpg" width="450px"> </div><br>

### Consistency

Consistency refers to the characteristic of whether multiple data copies can remain consistent. Under consistency conditions, the system can move from a consistent state to another consistency state after performing a data update operation.

After a data update to the system is successful, if all users can read the latest value, the system is considered to have strong consistency.

### Availability

Availability refers to the ability of a distributed system to provide normal services in the face of various exceptions. It can be measured by the ratio of the system's available time to the total time. The availability of four nines means that the system is available 99.99% of the time.

Under availability conditions, the services provided by the system are required to be always available, and results can always be returned within a limited time for each user operation request.

### Partition tolerance

Network partitioning means that nodes in a distributed system are divided into multiple areas. Each area can communicate within it, but cannot communicate between areas.

Under the condition of partition tolerance, the distributed system still needs to be able to provide external consistency and availability services when encountering any network partition failure, unless the entire network environment fails.

### Trade-offs

In distributed systems, partition tolerance is essential because one needs to always assume that the network is unreliable. Therefore, the CAP theory is actually a trade-off between availability and consistency.

Availability and consistency are often in conflict, and it is difficult to satisfy them at the same time. When synchronizing data between multiple nodes,

- In order to ensure consistency (CP), nodes that have not been synchronized cannot be accessed, thus losing part of the availability;
- To ensure availability (AP), data from all nodes is allowed to be read, but the data may be inconsistent.

## 4. BASE

BASE is the abbreviation of the three phrases Basically Available, Soft State and Eventually Consistent.

BASE theory is the result of the trade-off between consistency and availability in CAP. Its core idea is: even if strong consistency cannot be achieved, each application can adopt appropriate methods according to its own business characteristics to make the system achieve
to eventual consistency.


### Basically available

It means that when a distributed system fails, the core is guaranteed to be available and partial availability is allowed to be lost.

For example, when e-commerce is conducting promotions, in order to ensure the stability of the shopping system, some consumers may be directed to a downgraded page.

### Soft state

It means that the data in the system is allowed to exist in an intermediate state, and it is considered that the intermediate state will not affect the overall availability of the system, that is, there is a delay in the process of synchronizing data copies between different nodes in the system.

###Eventual consistency

Eventual consistency emphasizes that all data copies in the system can eventually reach a consistent state after a period of synchronization.

ACID requires strong consistency and is usually used in traditional database systems. BASE requires eventual consistency and achieves availability by sacrificing strong consistency. It is usually used in large distributed systems.

In actual distributed scenarios, different business units and components have different consistency requirements, so ACID and BASE are often used together.

## 5. Paxos

Used to reach consensus problems, that is, for values generated by multiple nodes, this algorithm can ensure that only a unique value is selected.

There are three main types of nodes:

- Proposer: Propose a value;
- Acceptor: votes on each proposal;
- Learner: is informed of the voting results and does not participate in the voting process.

#### 2. Accept stage

When a Proposer receives more than half of the Acceptor's Prepare responses, it can send an Accept request.

After Proposer A receives two Prepare responses, it sends [n=2, v=8] Accept request. This Accept request will be discarded by all Acceptors, because at this time all Acceptors guarantee not to accept proposals with sequence numbers less than 4.

Proposer B later also received two Prepare responses, so it also started sending Accept requests. It should be noted that v in the Accept request needs to be the v value corresponding to the largest proposal number it has received, which is 8. So it sends an Accept request for [n=4, v=8].

#### 3. Learn stage

When the Acceptor receives the Accept request, if the sequence number is greater than or equal to the minimum sequence number promised by the Acceptor, then the Learn proposal is sent to all Learners. When the Learner finds that a majority of Acceptors have accepted a certain proposal, the proposal value of the proposal is selected by Paxos.

### Constraints

#### 1\. Correctness

Indicates that only one proposed value will take effect.

Because the Paxos protocol requires that each valid proposal be accepted by a majority of acceptors, and acceptors will not accept two different proposals, correctness can be guaranteed.

#### 2\. Terminability

It means that there will always be a proposal that takes effect in the end.

The Paxos protocol can make the proposal sent by the Proposer move closer to the proposal that can be accepted by the majority of Acceptors, thus ensuring terminability.

### Single Candidate election

There are three types of nodes: Follower, Candidate and Leader. The Leader will periodically send heartbeat packets to the Follower. Each Follower is set with a random election timeout, generally 150ms\~300ms. If it does not receive the heartbeat packet from the Leader within this time, it will become a Candidate and enter the election stage.
