The network layer only sends the packet to the destination host, but the real communication is not the host but the process in the host. The transport layer provides logical communication between processes. The transport layer shields the core details of the underlying network layer from high-level users, making the application look like there is an end-to-end logical communication channel between the two transport layer entities.

## Characteristics of UDP and TCP

- User Datagram Protocol UDP (User Datagram Protocol) is connectionless, delivered as much as possible, has no congestion control, is message-oriented (the messages passed by the application are not merged or split, just add UDP headers), and supports one-to-one, one-to-many, many-to-one and many-to-many interactive communications.

- Transmission Control Protocol TCP (Transmission Control Protocol) is connection-oriented, provides reliable delivery, has flow control, congestion control, provides full-duplex communication, and is byte stream-oriented (treating messages from the application layer as a byte stream, and organizing the byte stream into data blocks of varying sizes). Each TCP connection can only be point-to-point (one-to-one).

## TCP header format

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/55dc4e84-573d-4c13-a765-52ed1dd251f9.png" width="700"/> </div><br>

- **Sequence Number**: Used to number the byte stream. For example, the sequence number is 301, which means the number of the first byte is 301. If the length of the data carried is 100 bytes, then the sequence number of the next segment should be 401.

- **Confirmation number**: The sequence number of the next segment expected to be received. For example, B correctly receives a message segment sent by A. The sequence number is 501 and
the data length is 200 bytes. Therefore, B expects the sequence number of the next message segment to be 701. The confirmation number in the confirmation message segment sent by B to A is 701.

- **Data offset**: refers to the offset of the data part from the beginning of the message segment. It actually refers to the length of the header.

- **Confirm ACK**: When ACK=1, the confirmation number field is valid, otherwise it is invalid. TCP stipulates that after the connection is established, all transmitted segments must have ACK set to 1.

- **Synchronization SYN**: Used to synchronize the sequence number when the connection is established. When SYN=1 and ACK=0, it indicates that this is a connection request segment. If the other party agrees to establish the connection, SYN=1 and ACK=1 in the response message.

- **Termination FIN**: Used to release a connection. When FIN=1, it means that the data of the sender of this segment has been sent and the connection is required to be released.

- **Window**: The window value serves as the basis for the receiver to let the sender set its sending window. The reason for this restriction is that the receiver's data cache space is limited.

## TCP three-way handshake

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/e92d0ebc-7d46-413b-aec1-34a39602f787.png" width="600"/> </div><br>

Assume A is the client and B is the server.

- First, B is in the LISTEN state, waiting for the client's connection request.

- A sends a connection request message to B, SYN=1, ACK=0, and selects an initial sequence number x.

- B receives the connection request message. If it agrees to establish the connection, it sends a connection confirmation message to A, with SYN=1, ACK=1, the confirmation number is x+1, and also selects an initial sequence number y.

- After A receives B's connection confirmation message, it also sends a confirmation to B. The confirmation number is y+1 and the sequence number is x+1.

- After B receives confirmation from A, the connection is established.

**The reason for the three-way handshake**

The third handshake is to prevent invalid connection requests from reaching the server and causing the server to open the connection incorrectly.

If the connection request sent by the client is stuck in the network, it will take a long time to receive the connection confirmation sent back by the server. After the client waits for a timeout retransmission period, it will re-request the connection. But this stuck connection request will eventually reach the server. If the three-way handshake is not performed, the server will open two connections. If there is a third handshake, the client will ignore the connection confirmation sent by the server later for the stuck connection request, and will not perform the third handshake, so the connection will not be opened again.

The following description does not discuss sequence numbers and confirmation numbers because the rules for sequence numbers and confirmation numbers are relatively simple. And ACK is not discussed because ACK is always 1 after the connection is established.

- A sends a connection release message, FIN=1.

- B sends a confirmation after receiving it. At this time, TCP is in a semi-closed state. B can send data to A but A cannot send data to B.

- When B no longer needs the connection, send a connection release message with FIN=1.

- A sends a confirmation after receiving it, enters the TIME-WAIT state, and waits for 2 MSL (maximum message survival time) before releasing the connection.

- B releases the connection after receiving A's confirmation.

**Reasons for waving four times**

After the client sends the FIN connection release message, the server receives the message and enters the CLOSE-WAIT state. This state is for the server to send data that has not yet been transferred. After the transfer is completed, the server will send a FIN connection release message.

**TIME_WAIT**

The client enters this state after receiving the FIN message from the server. At this time, it does not directly enter the CLOSED state. It also needs to
To wait for a time the timer is set to 2MSL. There are two reasons for this:

- Ensure that the last confirmation message can arrive. If B does not receive the confirmation message from A, it will resend the connection release request message. A waits for a period of time to handle this situation.

- Waiting for a period of time is to allow all packets generated within the duration of this connection to disappear from the network, so that the next new connection will not have old connection request packets.

## TCP sliding window

The window is part of the cache and is used to temporarily store the byte stream. The sender and receiver each have a window. The receiver tells the sender its window size through the window field in the TCP segment. The sender sets its own window
size based on this value and other information.

All bytes within the sending window are allowed to be sent, and all bytes within the receiving window are allowed to be received. If the bytes on the left side of the sending window have been sent and confirmed, then slide the sending window a certain distance to the right until the first byte on the left is no longer sent and confirmed; the sliding of the receiving window is similar. If the bytes on the left side of the receiving window have been sent and confirmed and delivered to the host, slide the receiving window to the right.

The receiving window will only acknowledge the last byte that arrives in sequence within the window. For example, the bytes received by the receiving window are {31, 34, 35}, of which {31} arrives in sequence, but {34, 35} does not, so only byte 31 is acknowledged. After the sender gets an acknowledgment of a byte, it knows that all bytes before this byte have been received.

## TCP flow control

Flow control is to control the sending rate of the sender to ensure that the receiver has time to receive.

The window field in the confirmation message sent by the receiver can be used to control the sender's window size, thereby affecting the sender's sending rate. Set the window field to 0 and the sender cannot send data.

## TCP congestion control

If the network is congested, packets will be lost and the sender will continue to retransmit, resulting in greater network congestion. Therefore, when congestion occurs, the sender's rate should be controlled. This is very similar to flow control, but the starting point is different. Flow control is to allow the receiver to receive in time, while congestion control is to reduce the congestion level of the entire network.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/51e2ed95-65b8-4ae9-8af3-65602d452a25.jpg" width="500"/> </div><br>

TCP mainly uses four algorithms to control congestion: slow start, congestion avoidance, fast retransmission, and fast recovery.

The sender needs to maintain a state variable called the congestion window (cwnd). Note the difference between the congestion window and the sender window: the congestion window is just a state variable, and it is the sender window that actually determines how much data the sender can send.

For the convenience of discussion, the following assumptions are made:

- The receiver has a large enough receive buffer so that flow control does not occur;
- Although TCP's window is based on bytes, the window size unit here is set to message segments.

### 1. Slow start and congestion avoidance

The initial execution of sending st
arts slowly. Let cwnd = 1. The sender can only send 1 segment; when the confirmation is received, cwnd is doubled, so the number of segments that the sender can send is: 2, 4, 8...

Note that slow start doubles cwnd every round, which will make cwnd grow very fast, causing the sender's sending speed to grow too fast, and the possibility of network congestion is higher. Set a slow start threshold ssthresh. When cwnd \>= ssthresh, congestion avoidance is entered, and only cwnd is increased by 1 in each round.

If a timeout occurs, set ssthresh = cwnd / 2 and re-execute slow start.

### 2. Fast retransmission and fast recovery

On the receiving side, it is required that each time a message segment is received, the last received ordered message segment should be acknowledged. For example, M<sub>1</sub> and M<sub>2</sub> have been received, and when M<sub>4</sub> is received, a confirmation of M<sub>2</sub> should be sent.

On the sender side, if three duplicate acknowledgments are received, it is known that the next segment is lost. At this time, fast retransmission is performed and the next segment is retransmitted immediately. For example, if three M<sub>2</sub> are received, M<sub>3</sub> will be lost and M<sub>3</sub> will be retransmitted immediately.

In this case, only individual segments are lost, not network congestion. Therefore, fast recovery is performed, letting ssthresh = cwnd / 2, cwnd = ssthresh. Note that congestion avoidance is directly entered at this time.

The speed of slow start and fast recovery refers to the setting value of cwnd, not the growth rate of cwnd. Slow start cwnd is set to 1, and fast recovery cwnd is set to ssthresh.
