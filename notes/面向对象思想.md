## 1. Three major characteristics

### Encapsulation

Use abstract data types to encapsulate data and data-based operations to form an indivisible independent entity. The data is protected inside the abstract data type, hiding the internal details as much as possible, and only retaining some external interfaces to connect with the outside. Users do not need to care about the internal details of the object, but they can access the object through the interface provided by the object.

Advantages:

- Reduce coupling: can be developed, tested, optimized, used, understood and modified independently
- Reduce the burden of maintenance: it can be easier to understand, and it can be debugged without affecting other modules
- Effectively tune performance: Profiling can be used to determine which modules impact system performance
- Improve software reusability
- Reduces the risk of building large systems: even if the entire system is unavailable, these independent modules may be available

The following Person class encapsulates attributes such as name, gender, and age. The outside world can only obtain the name attribute and gender attribute of a Person object through the get() method, but cannot obtain the age attribute. However, the age attribute can be used by the work() method.

Note that the gender attribute is stored using the int data type, and encapsulation makes this implementation detail invisible to users. And when you need to modify the data type used by the gender attribute, you can do it without affecting the client code.

```java
public class Person {

    private String name;
    private int gender;
    private int age;

    public String getName() {
        return name;
    }

    public String getGender() {
        return gender == 0 ? "man" : "woman";
    }

    public void work() {
        if (18 <= age && age <= 50) {
            System.out.println(name + " is working very hard!");
        } else {
            System.out.println(name + " can't work any more!");
        }
    }
}
```

### Inheritance

Inheritance implements the **IS-A** relationship. Fo
r example, Cat and Animal are an IS-A relationship, so Cat can inherit from Animal and obtain the non-private properties and methods of Animal.

Inheritance should follow the Liskov substitution principle, and subclass objects must be able to replace all parent class objects.

Cat can be used as Animal, which means you can use Animal to reference Cat objects. Parent class reference pointing to subclass object is called **upcast**.

```java
Animal animal = new Cat();
```

### Polymorphism

Polymorphism is divided into compile-time polymorphism and run-time polymorphism:

- Compile-time polymorphism mainly refers to method overloading
- Runtime polymorphism means that the specific type pointed to by the object reference defined in the program is determined during runtime

There are three conditions for runtime polymorphism:

- inheritance
- Overwrite (rewrite)
- Upward transformation

In the code below, the instrument class (Instrument) has two subclasses: Wind and Percussion. They both override the play() method of the parent class, and use the parent class Instrument to refer to the Wind and Percussion objects in the main() method. When the play() method is called on an Instrument reference, the play() method of the class where the actual reference object is located will be executed, not the method of the Instrument class.

```java
public class Instrument {

    public void play() {
        System.out.println("Instument is playing...");
    }
}
```

```java
public class Wind extends Instrument {

    public void play() {
        System.out.println("Wind is playing...");
    }
}
```

```java
public class Percussion extends Instrument {

    public void play() {
        System.out.println("Percussion is playing...");
    }
}
```

```java
public class Music {

    public static void main(String[] args) {
        List<Instrument> instruments = new ArrayList<>();
        instruments.add(new Wind());
        instruments.add(new Percussion());
        for(Instrument instrument : instruments) {
            instrument.play();
        }
    }
}
```

### Composition

Different from aggregation, the whole and the parts in the combination are strongly dependent. If the whole does not exist, the parts will not exist either. For example, companies and departments. A company would not exist without departments. But the company and its employees have an aggregation relationship, because the company is gone and the employees are still there.

### Association

Indicates that there is an association between objects of different types. This is a static relationship that has nothing to do with the status of the running process and can be determined at the beginning. Therefore, it can also be represented by 1-to-1, many-to-1, and many-to-many relationships. For example, students and schools are related. A school can have many students, but a student only belongs to one school. Therefore, this is a many-to-one relationship that can be determined before the operation starts.

### Dependency

Unlike associations, dependencies work during operation. There are three main forms of dependency relationships between classes A and B:

- Class A is a local variable of class B method;
- Class A is the parameter of class B method;
- Class A sends a message to class B, thereby affecting changes in class B.

## 3. Design principles

### S.O.L.I.D.

| Abbreviation | Quanpin | Chinese Translation |
| :---: | :---: | :---: |
| SRP | The Single Responsibility Principle | Single Responsibility Princ
iple |
| OCP | The Open Closed Principle | The Open Closed Principle |
| LSP | The Liskov Substitution Principle | Liskov Substitution Principle |
| ISP | The Interface Segregation Principle | The Interface Segregation Principle |
| DIP | The Dependency Inversion Principle | Dependency Inversion Principle |

#### 1. Single responsibility principle

> There should be only one reason to modify a class.

In other words, let a class be responsible for only one thing. When this class needs to do too many things, you need to decompose this class.

If a class assumes too many responsibilities, it is equivalent to coupling these responsibilities together. Changes in one responsibility may weaken the class's ability to complete other responsibilities.

#### 2. Open and closed principle

> Classes should be open for extension and closed for modification.

Extension means adding new functionality
thinking, so this principle requires that new functionality be added without modifying the code.

The most typical design pattern that conforms to the open-closed principle is the decorator pattern, which can dynamically attach responsibilities to objects without modifying the class code.

#### 3. Richter substitution principle

> Subclass objects must be able to replace all parent class objects.

Inheritance is an IS-A relationship. The subclass needs to be able to be used as the parent class and needs to be more special than the parent class.

If this principle is not met, the behavior of each subclass will be very different, increasing the complexity of the inheritance system.

#### 4. Interface separation principle

> Clients should not be forced to rely on methods they don't use.

Therefore it is better to use multiple specialized interfaces than a single overall interface.

#### 5. Dependency inversion principle

>High-level modules should not depend on low-level modules, both should depend on abstraction;</br>Abstraction should not depend on details, details should depend on abstraction.

The high-level module contains important strategic choices and business modules in an application. If the high-level module depends on the low-level module, changes in the low-level module will directly affect the high-level module, forcing the high-level module to also need to be changed.

Dependent on abstraction means:

- No variable should hold a pointer or reference to a concrete class;
- No class should be derived from a concrete class;
- No method should override an already implemented method in any of its base classes.

### Other common principles

In addition to the above classic principles, there are also the following common design principles in actual development.

| Abbreviation | Quanpin | Chinese Translation |
| :---: | :---: | :---: |
|LOD| The Law of Demeter | The Law of Demeter |
|CRP| The Composite Reuse Principle | The Composite Reuse Principle |
|CCP| The Common Closure Principle | Common Closure Principle |
|SAP| The Stable Abstractions Principle | Sta
ble Abstractions Principle |
|SDP| The Stable Dependencies Principle | Stable Dependencies Principle |

#### 1. Demeter’s Law

Demeter's Law is also called the Least Knowledge Principle (LKP), which means that an object should know as little as possible about other objects and should not talk to strangers.

#### 2. Principle of synthesis and reuse

Try to use object composition instead of inheritance to achieve reuse.

#### 3. Common closure principle

Classes that are modified together should be grouped together (in the same package). If we must modify the code in the application, we want all modifications to occur in one package (modify closed), rather than spread across many packages.

#### 4. Stable abstraction principle

The most stable package should be the most abstract package, and the unstable package should be a concrete package. That is, the degree of abstraction of a package is directly proportional to its stability.

#### 5. Stable dependency principle

The dependencies between packages should be in a stable direction, and the packages that a package depends on are more stable than itself.
