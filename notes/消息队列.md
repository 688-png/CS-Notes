## 1. Message model

### Peer-to-peer

After the message producer sends a message to the message queue, it can only be consumed once by one consumer.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191212011250613.png"/> </div><br>

### Publish/Subscribe

After a message producer sends a message to a channel, multiple consumers can subscribe to the message from the channel and consume it.

<div align="center"> <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/image-20191212011410374.png"/> </div><br>

The publish-subscribe pattern and the observer pattern have the following differences:

- In the observer pattern, both the observer and the subject know the existence of each other; in the publish and subscribe pattern, the producer and the consumer do not know the existence of each other, and they communicate through channels.
- The observer mode is synchronous. When an event is triggered, the subject will call the observer's method and then wait for the method to return; while the publish and subscribe mode is asynchronous. After the producer sends a message to the channel, it does not need to care when the consumer subscribes to the message and can return immediately.

## 2. Usage scenarios

### Asynchronous processing

After the sender sends the message to the message queue, it does not need to wait synchronously for the message receiver to complete processing, but immediately returns to perform other operations. The message receiver subscribes to the message from the message queue and processes it asynchronously.

For example, during the registration process, it is usually necessary to send a verification email to ensure the legitimacy of the registered user's identity. You can use the message queue to make the operation of sending the verification email asynchronous. The user can complete the registration after filling in the registration information, and the message of sending the verification email is sent to the message queue.

This can only be done if the business process allows asynchronous processing. For example, in the above registration process, if the user is required to click on the verification
email before completing the registration, the message queue can no longer be used.

### Traffic sharpening

In a high-concurrency scenario, if a large number of requests arrive in a short period of time, the server will be overwhelmed.

Requests can be sent to the message queue, and the server subscribes to messages from the message queue for processing according to its processing capabilities.

### Application decoupling

If modules do not call each other directly, the coupling between modules will be very low, so modifying one module or adding a new module will have little impact on other modules, thereby achieving scalability.

By using the message queue, one module only needs to send messages to the message queue, and other modules can selectively subscribe to messages from the message queue to complete the call.

## 3. Reliability

### Reliability of the sending end

After the sending end completes the operation, the message must be successfully sent to the message queue.

Implementation method: Create a message table in the local database, and save the message data and business data in the same database instance, so that the transaction mechanism of the local database can be used. After the transaction is submitted successfully, the message in the message table is transferred to the message queue. If the message is transferred successfully, the data in the message table is deleted, otherwise retransmission continues.

### Reliability of the receiving end

The receiving end can successfully consume a message from the message queue.

Two implementation methods:

- Ensure that the business logic of message processing at the receiving end is idempotent: as long as it is idempotent, the final processing result will be the same no matter how many times the message is consumed.
- Ensure that messages have unique numbers and use a log table to record the message numbers that have been consumed.
